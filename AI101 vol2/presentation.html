Commands

Команды — язык управления агентом: режимы, контекст и готовые сценарии в одно действие.
Команды · что это

Что такое
команды?

Команды — это то, как вы обращаетесь к агенту: переключаете режимы, добавляете контекст, запускаете готовые сценарии.

Вы
→
/compress  управление сеансом
@api.md  файл в контекст
!git status  вывод оболочки
/git:commit  мини-сценарий
→
Агент
получает
задачу
Команды · обзор

Три входа
и один паттерн

/

Управление GigaCode: сеанс, инструменты, навыки, модель, настройки.

@

Добавить файл или директорию в текущий разговор.

!

Выполнить shell-команду или перейти в режим оболочки.

свои

Markdown-сценарий запроса с параметрами и вставками.

Команды через /

Slash-команды:
категории и режимы

КатегорияКомандыЗачем
Сеанс/init, /compress, /resume, /recap, /contextУправлять памятью и состоянием разговора.
Настройка/mcp, /tools, /skills, /approval-mode, /modelВидеть и менять окружение агента.
Безопасность/restore, /clear, /quitОткат, чистый старт, выход.
@ и !

Контекст
и оболочка рядом

@ кладёт факты в разговор

  • @docs/api.md добавляет файл.
  • @src/auth добавляет директорию с учётом правил чтения.
  • Главный риск: слишком широкий контекст.

! запускает оболочку

  • !git status показывает реальное состояние.
  • Одиночный ! переводит в режим оболочки.
  • В подпроцессе выставляется GIGACODE=1.
Пользовательские команды

Markdown как
мини-сценарий

  • Личные: ~/.gigacode/commands/
  • Проектные: .gigacode/commands/
  • Проектные команды имеют приоритет.
  • Подпапка задаёт префикс: git/commit.md → /git:commit.
# .gigacode/commands/git/commit.md description: создать сообщение коммита Проверь изменения в индексе: !{git diff --staged} Напиши сообщение коммита для: {{args}}
Когда использовать

Команды хороши,
когда нужна точка входа

Повторяемость

Частый сценарий, который не хочется каждый раз объяснять заново.

Контекст

Нужно добавить конкретные файлы без длинной преамбулы.

Состояние

Нужно быстро узнать положение дел командой оболочки.

Репозиторий

Сценарий лежит в git и обновляется вместе с кодом.

Лёгкость

Готовый запрос уже полезен, а полноценный навык пока избыточен.

Чек-лист

Обязательные шаги и параметры зашиты в самой команде — никто не пропустит.

Проблемы

Что ломается,
когда команд становится много

  • Десятки своих команд превращаются в неявный интерфейс — без справки и навигации.
  • !{...} внутри команды — это код, который кто-то должен проверять.
  • Длинная команда — это процесс. Без ревью он стареет молча.
  • @directory чаще добавляет шум, чем контекст.

Инструменты · что это

Что такое
инструменты?

Инструмент — конкретное действие агента. Вы задаёте, что требует подтверждения, — через режим подтверждений.

Агент
решает
→

Факт получен,

Как работает вызов

Шаги вызова
инструмента

01

Выбор

Модель читает описания инструментов и решает, какой подходит для следующего шага.

02

Параметры

Формирует структурированный вызов: имя инструмента и аргументы.

03

Контроль

Режим подтверждений и хуки решают: разрешить, спросить или заблокировать.

04

Результат

Вывод инструмента возвращается в разговор как новый факт.

Навыки учат агента как думать. Инструменты дают ему действия и факты.

Типы инструментов

Шесть семейств
инструментов

Поиск

grep_search, glob, list_directory — найти нужное место в коде.

Чтение

read_file, read_many_files — забрать содержимое в контекст.

Правка

edit для точечных изменений, write_file для создания или полной замены.

Оболочка

run_shell_command — тесты, сборка, git, любые CLI-инструменты.

Сеть

web_fetch — взять документацию, OpenAPI-схему, внешнюю спецификацию.

Память

save_memory, todo_write — зафиксировать факты и план задачи.

Ещё бывают MCP-инструменты — это блок 5; делегирование субагенту — блок 6.

Управление

Режим подтверждений —
граница автономности

РежимЧто разрешаетРиск
planАнализ без исполнения.Безопасно, но без действий.
defaultПодтверждение для правок и команд оболочки.Хороший режим на каждый день.
auto-editАвтоподтверждение для инструментов правки.Удобно в доверенной рабочей папке.
yoloАвтоподтверждение всего.Быстро, но широкая зона последствий.

Переключение: /approval-mode plan · default · auto-edit · yolo. Без аргумента команда показывает текущий режим.

Политика

Под каждый тип —
своя политика

  • Чтение и поиск — без подтверждений, минимальный риск.
  • Правки — подтверждение в default; auto-edit только в доверенной директории.
  • Оболочка — белый список команд плюс PreToolUse hook на опасные паттерны.
  • Сетевые запросы (web_fetch) — отдельная политика для приватных URL и внешних сервисов.
  • Деструктивные действия (rm, git push --force) — PreToolUse с требованием явной цели.
Проблемы

Инструменты:
польза и издержки

Уносят контекст
Большой вывод

Объёмный результат забивает окно — модель теряет фокус задачи.

Лишние вызовы

Цепочка обращений без прогресса — токены тратятся, ответ откладывается.

Расширяют доступ
Оболочка

Команды зависят от окружения и могут быть деструктивными.

YOLO-режим

Граница согласований исчезает целиком — любое действие проходит.

Сетевые запросы

web_fetch уносит параметры и URL во внешний сервис — не для приватных систем.

Навыки · обзор

Навыки — упакованная
экспертиза через SKILL.md

Инструкции

Как думать, какие инварианты держать, какие шаги не пропускать.

Скрипты

Локальные вспомогательные скрипты, которые навык может рекомендовать запускать.

Шаблоны

Готовые формы отчётов, миграций, проверок и заметок к выпуску.

Материалы

Вспомогательные файлы с предметной экспертизой.

Описание

Главный триггер — модель решает использовать навык по его описанию.

Область

Узкий навык лучше широкого справочника обо всём.

Где лежат

Три уровня
подключения

Личные

~/.gigacode/skills/
Персональные привычки и повторяемые задачи.

Проект

.gigacode/skills/
Командные стандарты, которые можно версионировать.

Расширение

Навыки, поставляемые расширениями и общими пакетами.

Подключение: автоматически — по description; вручную — через /skills.

Формат

Что внутри
SKILL.md

--- name: java-pr-review description: Проверка Java-запроса на слияние: код, тестовые риски, миграции. paths: - "src/**/*.java" --- 1. Сначала прочитать изменения. 2. В первую очередь искать ошибки и регрессии. 3. Указывать файл и строку. 4. Отмечать пробелы в тестах.
  • name — стабильный идентификатор.
  • description — формулировки, которыми пользователь описывает задачу.
  • paths — помогает найти навык, но не ограничивает доступ.

Проблемы

Расплывчатый навык
срабатывает везде — и мешает

  • Широкий description включает навык слишком часто.
  • Несколько навыков могут конфликтовать в инструкциях.
  • Длинный навык съедает контекст вместо того, чтобы помогать.
  • Устаревший проектный навык ломает процесс всей команды.

Hooks

Hooks, или хуки, делают поведение агента предсказуемым: автоматически срабатывают на события.
Hooks · что это

Что такое
hooks?

Hook — скрипт или HTTP-вызов, который запускается автоматически при событии: перед действием агента, после него или при старте сессии.

Событие
PreToolUse,
SessionStart…
→
Hook
в фоне
невидим для модели
→
✓ разрешить
⊘ заблокировать
◎ записать в лог

Hooks дают детерминированный контроль там, где решение нельзя оставлять на усмотрение модели.

Структура hook

Из чего состоит
hook

# ~/.gigacode/settings.json или .gigacode/settings.json "hooks": { "PreToolUse": [ { "matcher": "run_shell_command", "hooks": [ { "type": "command", "command": "./scripts/safety.sh" } ] } ] }
  • event — что произошло: PreToolUse, SessionStart и т.д.
  • matcher — regex по имени инструмента или субагента.
  • type — command (скрипт), http (URL), function (внутренний).
  • Лежит в settings.json: личный, проектный или из расширения.
События

Где можно
использовать

ГруппаСобытияТипичный сценарий
ИнструментыPreToolUse, PostToolUse, PostToolUseFailureПроверка параметров, аудит, обработка ошибок.
ВводUserPromptSubmit, PermissionRequestДоб. контекста к запросу, политика выдачи прав.
СеансSessionStart, SessionEnd, Stop, NotificationИнициализация, очистка, уведомления.
ПамятьPreCompact, PostCompactСохранить или дополнить контекст до сжатия.
ДелегированиеSubagentStartПодготовка контекста для субагента.
Исполнители

Два рабочих
типа: command и http

command

Локальный скрипт. JSON приходит через stdin, ответ — через stdout.

"type": "command", "command": "./scripts/safety.sh", "timeout": 5000

http

POST JSON на URL. Подходит для внешнего сервиса политики или аудита.

"type": "http", "url": "https://audit/hook", "headers": { … }

Есть ещё function — для встроенных механизмов; в пользовательских настройках почти не встречается.

Тонкая настройка

Как срабатывает
и что возвращает

matcher

Regex по имени инструмента или субагента. Широкий regex — частая причина лишних блокировок.

parallel · sequential

По умолчанию параллельно. sequential: true — цепочка; следующий хук видит вывод предыдущего.

async

async: true — не блокирует агента, результат вернётся следующим ходом. Для аудита и метрик.

Exit codes

0 — успех (читаем JSON из stdout). 2 — блокирующая ошибка (stderr идёт в агента). Другое — non-blocking.

PreToolUse ответ

Обязательные поля: permissionDecision (allow/deny/ask) и permissionDecisionReason.

Аварийный выключатель

disableAllHooks: true — снять весь слой, если что-то сломалось.

Проблемы → решения

Где hooks ломаются
и что делать

ПроблемаРешение
Медленный hook тормозит каждый вызов инструмента.async: true для аудита; timeout для command; sequential — только для критичных проверок.
Пользователь не понимает, почему действие заблокировано.Заполнять permissionDecisionReason читаемой фразой — она показывается в терминале.
HTTP hook может стучаться куда угодно.allowedUrls и allowedEnvVars; SSRF-защита по DNS встроена.
Hook сломан — агент стоит.disableAllHooks: true в личном settings.json — аварийный выключатель.

MCP

MCP открывает агенту внешний мир: инструменты, базы данных и API — через единый протокол.
MCP · что это

Что такое
MCP?

MCP (Model Context Protocol) — стандарт для подключения внешних инструментов и данных. Агент работает с ними так же, как со встроенными.

Jira / Linear
База данных
Confluence / Docs
Внутренний API
↔
MCP
сервер
адаптер
↔
Агент
видит инструменты
и данные

Вы контролируете, какие инструменты открыты модели. Доступ на запись закрывается отдельно.

MCP-сервер

Что такое
MCP-сервер

Сервер — не «отдельная машина», а процесс, который говорит на MCP-протоколе. Один сервер = один пакет инструментов (Jira, БД, документация).

Локальный (stdio)

Дочерний процесс запускается рядом с агентом. Никакой сети. Подходит для CLI, доступа к локальной БД, кастомных серверов.

Удалённый (http)

HTTP-сервис, к которому подключается агент. Подходит для корпоративных Jira, GitHub, Confluence, внутренних API.

Свой сервер пишем, только если нет готового. Для популярных систем серверы уже есть — подключаем готовые.

Проблемы

Что MCP забирает
за свою пользу

  • Каждый инструмент сервера попадает в каждый запрос модели — растёт длина контекста и падает точность выбора.
  • Внешний сервер видит параметры запросов и доступ — для приватных систем это требует отдельной оценки.
  • OAuth-токены протухают молча: инструмент исчезает, агент действует так, будто его никогда не было.
  • trust: true стирает границу подтверждений — все вызовы инструментов сервера идут без вопросов.

Subagents

Субагенты берут часть задачи на себя — каждый со своей областью, контекстом и инструментами.
Субагенты · что это

Что такое
субагенты?

Субагент — отдельный экземпляр агента, который берёт на себя часть задачи и возвращает результат главному агенту.

Главный
агент
→
Именованный субагент
блокирует родителя · начинает с чистого листа
Fork-субагент
работает в фоне · наследует контекст
→
Результат
или
статус

Именованный субагент возвращает результат в главный разговор; fork показывает прогресс, но не подмешивает результат автоматически.

Субагенты · обзор

Специализированные
ИИ-помощники

Задание

У каждого может быть свой системный промпт и стиль работы.

Инструменты

Можно задать список разрешённых или запрещённых инструментов.

Контекст

Отдельный контекст разговора, результат возвращается главному агенту.

Именованный

Блокирует родителя до завершения. Начинает с чистого контекста — без истории родителя.

Fork-субагент

Работает в фоне. Наследует полный контекст родителя. Родитель продолжает работу.

Конфигурация

Хранится в .gigacode/agents/ (проект) или ~/.gigacode/agents/ (личное).

Анатомия субагента

Из чего состоит
именованный агент

--- name: code-reviewer description: второй взгляд на изменения, ищет регрессии и нарушения контракта model: inherit approvalMode: plan tools: - read_file - grep_search - glob disallowedTools: - edit - write_file - run_shell_command --- # Системный промпт Ты делаешь ревью изменений. Всегда ссылайся на file:line. Сначала ищи регрессии и нарушения контракта.
  • name — стабильный идентификатор для Task(subagent_type=…).
  • description — формулировки, по которым модель выбирает агента.
  • model: inherit — берёт модель родителя; можно задать конкретную.
  • approvalMode — свой режим подтверждений, обычно plan для read-only.
  • tools — белый список. Без него субагент наследует все инструменты, включая MCP.
  • Тело — системный промпт: как думать, что возвращать, какие правила держать.
  • Лежит в .gigacode/agents/ (проект) или ~/.gigacode/agents/ (личное). Управление — /agents.
Когда использовать

Субагент получает
узкую область задачи

Тестирование

Найти тестовые риски и сделать точечные проверки.

Документация

Собрать черновик по известному шаблону.

Безопасность

Проверка чувствительных мест только на чтение.

Рефакторинг

Исследовать последствия изменений без автоматических правок.

Ссылки на факты

Требовать файл/строку, команды и проверяемые данные.

Координация

Не давать нескольким агентам менять одни файлы.

Проблемы → решения

Где субагенты ломаются
и что делать

ПроблемаРешение
Субагент стартует с чистого листа — теряется контекст родителя.Передавать файлы, диффы и ссылки прямо в промпт Task(prompt=…).
По умолчанию наследует все инструменты, включая MCP с записью.Указывать tools: явно — белый список; для анализа добавлять approvalMode: plan.
Возвращает уверенное мнение без проверяемых ссылок.В промпте требовать file:line, команды и цитаты; в системном промпте — то же правило.
Несколько субагентов дублируют работу или спорят выводами.Разнести по областям (тесты / контракт / документация); не запускать на одних и тех же файлах.
