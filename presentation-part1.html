ЗАПРОС БЕЗ КОНТЕКСТА
> Почему refreshToken возвращает 500?
refreshToken может возвращать 500 из-за ошибки сервера. Проверьте логи и убедитесь, что endpoint корректно обрабатывает исключения. Возможно, истёк срок действия токена или есть проблемы с базой данных...

↑ Правдоподобно. Бесполезно.

ЗАПРОС С КОНТЕКСТОМ
> Почему refreshToken возвращает 500? [auth/token/refresh.js · middleware/auth.js] [тикет AUTH-441: ротация не работает]
В строке 47 refresh.js при ротации токена выбрасывается необработанное исключение: TokenStore.rotate() ожидает транзакцию, но middleware/auth.js вызывает его вне транзакционного контекста.

↑ Конкретно. Проверяемо.

Модель та же. Разница — в том, что попало в контекст. Этому посвящена вся первая часть.

БЛОК 2 · ТЕРМИНЫ

Где здесь LLM?

ОТ ШИРОКОГО К УЗКОМУ
  • AI — общий зонтик. OCR, рекомендации, планировщики маршрутов.
  • ML — система учится на данных. Скоринг, прогноз нагрузки, фрод.
  • Нейросети — один класс ML. CV-детекция, NLP-классификаторы.
  • LLM — большая языковая модель. Код, текст, анализ требований. Заметный класс, но не весь AI.
БЛОК 3 · LLM vs КЛАССИЧЕСКИЙ ML

Разные инструменты,
разные задачи

Классический ML LLM
Тип задачи узкая, заранее заданная широкий класс через язык
Вход фичи, структурированные данные контекст, инструкции, текст
Выход предсказанное число или класс текст / код / план / анализ
Метрика понятная, заранее определена часто субъективная, нужна проверка
Воспроизводимость высокая вероятностная — может варьироваться
Примеры фрод, скоринг, прогноз нагрузки код-ревью, анализ требований, документация
Когда НЕ LLM Стабильная массовая задача Когда нужна точная числовая метрика, миллионы запросов в день, воспроизводимый результат
ВЫВОД
LLM — не замена ML. Это другой инструмент для другого класса задач.
БЛОК 4 · ТИПЫ МОДЕЛЕЙ

Модели различаются по назначению и архитектуре

Не всё, что называют «LLM», устроено и применяется одинаково.

ПО НАЗНАЧЕНИЮ
CHAT / INSTRUCT

Диалог, инструкции, объяснения, черновики.

CODE

Код, repo-контекст, diff, тесты, зависимости.

EMBEDDING

Текст → вектор. Semantic search и RAG.

REASONING

Многошаговый анализ: баги, миграции, противоречия.

ПО АРХИТЕКТУРЕ
DENSE

Весь «мозг» модели участвует в каждом токене. Проще предсказать по latency и качеству.

[token] -> [full dense model] -> next token

MOE · MIXTURE OF EXPERTS

На каждый токен активируется часть «экспертов». Много параметров, но считается только часть — эффективнее.

[token] -> router -> [exp 2]+[exp 7] -> next token

«100B параметров» не значит «дороже в 10 раз» или «лучше во всём» — важны назначение, активные параметры, контекст, latency и tool use.

Как работает
нейросеть

Проследим путь одного запроса от текста до ответа. На реальном примере.
БЛОК 5 · ИНТУИЦИЯ

Нейросеть —
это функция

  • x — вход: текст, код, инструкция.
  • θ — параметры. Числа, настроенные на обучении.
  • f(x; θ) — выход: следующий токен, embedding, классификация.

В обычной программе поведение задаётся кодом.
В нейросети — архитектурой + параметрами.

Поэтому модель умеет обобщать, но её поведение менее прозрачно.

БЛОК 5 · ДВА РЕЖИМА

Обучение ≠ использование

ОБУЧЕНИЕ · TRAINING

Параметры настраиваются

  • Миллионы примеров → предсказания → ошибка → коррекция
  • Происходит до того, как мы получили модель
  • Очень дорого. Делается редко.
Параметры — числа внутри модели, которые задают, какие связи она считает сильными.
ИНФЕРЕНС · INFERENCE

Параметры применяются

  • Получили контекст → применили готовые θ → ответ
  • Происходит каждый раз, когда мы пишем в GigaCode
  • Модель не учится на нашем проекте в момент диалога
Во время ответа эти числа не меняются — модель только применяет уже настроенные связи к текущему контексту.
ПОЧЕМУ ЭТО ВАЖНО

Модель не «запоминает» наш репозиторий между сессиями. Если знание о проекте нужно — оно должно попасть в контекст.

БЛОК 5 · ШАГ 2 · EMBEDDINGS

Каждый токен —
точка в пространстве смыслов

  • Embedding — вектор из сотен или тысяч чисел.
  • В этом пространстве близкие по смыслу токены оказываются рядом.
  • refresh­Token, auth, JWT, 401 — попадут в один кластер.
  • 500, exception, stack trace — в соседний.
  • cooking, sunset — где-то далеко.

Это та же геометрия, на которой строится RAG: ищем похожие куски документов по близости в этом пространстве.

Где здесь параметры?
Параметры — настроенные числа внутри модели. Они не хранят «refreshToken = JWT», но влияют на то, что refreshToken, auth, JWT, 401 оказываются близко.
БЛОК 5 · ШАГ 3 · ATTENTION

Какие токены
смотрят на какие

УПРОЩЁННАЯ МАТРИЦА ВНИМАНИЯ
ЧТО ЗДЕСЬ ВИДНО
  • Между 500 и 401 — высокий вес связи: для модели это сигнал пары ошибок.
  • refresh и Token имеют высокую взаимную связь — сигнал единого концепта.
  • 401 активно связан с refresh­Token — для модели это сигнал auth-контекста.

Attention — механизм, который позволяет модели учитывать связи между токенами в контексте.

Длинный шумный контекст «размазывает» attention. Поэтому большое окно ≠ хороший ответ.

БЛОК 5 · ШАГ 4 · DECODING

Не выбирает ответ.
Строит токен за токеном.

ПОСЛЕ КОНТЕКСТА «функция refreshToken возвращает 500 вместо 401. ПОТОМУ ЧТО»

Модель оценивает распределение вероятностей следующего токена

REPEAT

Выбранный токен → в контекст. Цикл повторяется до конца ответа.

ВЫБОР

Модель не «знает истину». Она сэмплит следующий токен из распределения.

КАК УПРАВЛЯЕМ ВЫБОРКОЙ
Temperature

Резкость распределения. 0 — стабильно (для кода). Выше — разнообразнее.

Top-k

Только k самых вероятных токенов. k=3 → только «внутри», «обработ», «токен».

Top-p

Минимальный набор с суммарной вероятностью ≥ p. p=0.8 → берём верхние пока сумма ≥ 80%.

Управляют стилем выборки, не истинностью ответа.
БЛОК 5 · DECODING · ФИЛЬТРЫ

Top-k и Top-p:
два фильтра перед выбором токена

Модель уже посчитала вероятности. Теперь мы решаем, из каких кандидатов ей разрешено выбирать.

Сегодня на завтрак я съел...
TEMPERATURE

Меняет резкость распределения до сэмплинга.

t = 0.1

«кашу» становится почти очевидным выбором. Ответ стабильнее.

t = 1.2

Хвост вариантов получает больше шансов. Ответ разнообразнее.

МЕНЮ МОДЕЛИ
кандидатPсумма
кашу40%40%
яблоко30%70%
бутерброд15%85%
яичницу10%95%
пиццу3%98%
кирпич1%99%
облако1%100%
TOP-K = 3
  • ✓ кашу      40%
  • ✓ яблоко    30%
  • ✓ бутерброд 15%
  • ✕ яичницу
  • ✕ пиццу
  • ✕ кирпич
  • ✕ облако

Top-k берёт фиксированное число кандидатов.
k = 3 значит: «оставь только 3 самых вероятных».

TOP-P = 0.85
  • ✓ кашу      40% → сумма 40%
  • ✓ яблоко    30% → сумма 70%
  • ✓ бутерброд 15% → 85% STOP
  • ✕ яичницу
  • ✕ пиццу
  • ✕ кирпич
  • ✕ облако

Top-p берёт столько кандидатов, сколько нужно, чтобы набрать порог вероятности.
p = 0.85 значит: «оставь ядро вариантов, которое вместе даёт 85%».

Top-k режет по количеству. Top-p режет по суммарной вероятности. После этого модель сэмплит из оставшихся вариантов.

На практике эти фильтры управляют разнообразием ответа, но не проверяют его истинность.

БЛОК 6 · КОНТЕКСТ

Контекст — это всё,
что модель видит сейчас

  • Это не только наш последний запрос.
  • В контекст попадает всё: системный промпт, история, файлы, результаты tools, GIGACODE.md.
  • Окно ограничено — измеряется в токенах.
  • Большой контекст ≠ хороший контекст.

Качество ответа часто зависит не от того, какая у нас модель, а от того, что мы положили в контекст.

БЛОК 6 · РАЗГРАНИЧЕНИЕ

Три похожих слова

CONTEXT

То, что модель видит прямо сейчас в окне.

Например: текущий чат, прикреплённый файл, результат команды.

Исчезает после закрытия сессии.

MEMORY

Внешний механизм, который сохраняет факты между сессиями и подставляет их в контекст.

Например: «в этом проекте используем Maven», «клиент — банк, данные чувствительные».

RAG

Retrieval-Augmented Generation. Перед ответом ищем релевантные документы и добавляем в контекст.

Например: нашли 3 страницы из Confluence по запросу — положили в контекст.

Практически это один вопрос: что попадёт в окно модели перед ответом?

Ограничение в токенах. «128K context» — это 128K токенов, не символов.

Чем больше токенов на вход и выход — тем дольше ответ.

ВЫВОД

Шумный и раздутый контекст — дороже, медленнее, модель отвечает хуже. Инженерный аргумент за чистый и точный контекст.

БЛОК 8 · ПРОБЛЕМА #1

Почему «загрузи всё» ломает ответ

Проблема не только в размере — контекст может быть шумным, противоречивым или опасным.

1 · ШУМ

Всё подряд: старые docs, lock-файлы, нерелевантные тесты.
→ важные файлы теряются в потоке.

2 · ПЕРЕПОЛНЕНИЕ

Логи, JSON, stack traces быстро съедают окно.
→ модель теряет нужные детали.

3 · УСТАРЕВШИЙ

README, ADR, тикет описывают прошлое.
→ уверенный ответ про старую версию.

4 · КОНФЛИКТ ИНСТРУКЦИЙ

System prompt, rules, тикет и запрос говорят разное.
→ модель выбирает компромисс, который никто не просил.

5 · PROMPT INJECTION

Внешний документ содержит инструкции: «ignore previous rules».
→ данные конкурируют с правилами.

6 · CONTEXT DRIFT

Длинная сессия накапливает отменённые гипотезы и лишние outputs.
→ агент постепенно уходит от задачи.

Большой контекст ≠ хороший. Хороший — минимальный, актуальный, непротиворечивый, привязанный к задаче.

БЛОК 8 · РЕШЕНИЕ #1

Как собрать хороший контекст

1 · СУЗИТЬ SCOPE

Одна задача → минимальный набор файлов и фактов.

2 · ПРОВЕРИТЬ ИСТОЧНИКИ

Код и тесты показывают поведение. Тикеты и docs — намерение, но могут устареть.

3 · РАЗДЕЛИТЬ РОЛИ

Инструкции отдельно. Данные отдельно. Формат ответа отдельно.

4 · СЖАТЬ ШУМ

Логи, JSON, stack traces — в summary, если детали не нужны.

5 · ОТДЕЛИТЬ ВНЕШНИЕ ДАННЫЕ

Jira, wiki, письма — источник фактов, а не инструкции для агента.

6 · ЗАДАТЬ РЕЖИМ CLI-АГЕНТА

Только анализ, только план, правки после подтверждения или review-only.

Хороший контекст не максимальный — достаточный для задачи и проверяемый.

БЛОК 9 · ПРОБЛЕМА #2

Даже хороший контекст
можно испортить плохой задачей

ЗАПРОС
> Напиши документацию по фиче refresh-token
ЧТО ВЕРНЁТСЯ
# Документация: refreshToken refreshToken — метод обновления токена. Вызывается автоматически после истечения срока. Обращается к AuthService для проверки. При ошибке возвращает 500. Система поддерживает ротацию токенов. Рекомендуется хранить refresh_token в httpOnly cookie...

↑ Выглядит связно. Факты и домыслы перемешаны.

ПОЧЕМУ ЭТО ПРОИСХОДИТ

  • Структура каждый раз разная — нечего сравнивать
  • Открытые вопросы теряются в тексте
  • Режим не указан: анализ или сразу черновик?

Запрос — это не спецификация. Модель додумывает формат и критерии сама.

БЛОК 9 · РЕШЕНИЕ #2

Prompt — это
мини-спецификация

ШАБЛОН ХОРОШЕГО ЗАПРОСА
Цель: что нужно сделать Объект: конкретная фича / файл / задача Входные данные: что есть Режим: analysis-only / plan / draft Формат вывода: секции / таблица / файл Стоп-условие: когда НЕ делать следующий шаг
